What this guide is for

This guide helps your team recreate the same development motion used in the Trailer Prioritization work: clear business framing, practical prompts, focused implementation, validation after each meaningful change, and polished customer-ready output at the end.

1. One-time setup

A developer should be able to complete this once and be ready to start building in about 15 to 20 minutes.

What you need

  • A GitHub account with GitHub Copilot enabled
  • Visual Studio Code, latest stable version
  • Node.js LTS for web application work
  • Python 3.11 or later for Python and automation work
  • Access to the project repository in GitHub

Install in VS Code

  1. Install VS Code.
  2. Install these extensions: GitHub Copilot, GitHub Copilot Chat, ESLint, Prettier, and Python.
  3. Sign in to GitHub from VS Code.
  4. Open Copilot Chat and ask it to summarize the current folder to confirm it is active.
Recommended first terminal commands
git clone https://github.com/<your-org>/<your-repo>.git
cd <your-repo>

2. The vibe-coding workflow

This is the exact development loop to follow for each change. The point is not to ask Copilot for generic ideas. The point is to give it a clear outcome and keep momentum through fast validation.

1. Frame the business goal Start with the outcome. Example: reduce manual trailer scheduling effort and improve OTIF-driven prioritization.
2. Give Copilot a concrete task Include the files, users, constraints, and what “done” means.
3. Ask it to implement Prefer direct edits and validation over long explanations.
4. Validate immediately Run the build, tests, or app preview after each meaningful change.
5. Update docs in the same pass Keep setup notes, demo notes, and technical documentation aligned with the current behavior.
6. Package for the business End with one technical artifact and one business-friendly artifact.

3. Prompt templates your team can reuse

These are intentionally concrete. The better the scope, the better the result.

Feature build prompt
Implement [feature] in [path/file].
Context:
- Business goal: [goal]
- Users: [personas]
- Constraints: [performance/security/style]
Do the work end-to-end:
1) Edit code
2) Run validation
3) Update related docs
4) Summarize changes with file references
UI improvement prompt
Refactor this UI to be clearer for operations users.
Keep the same functionality, improve hierarchy, spacing, labels, and responsive behavior.
Then run a production build and report any issues fixed.
Data realism prompt
Expand local mock data so demos feel realistic.
Add enough records for meaningful KPIs and status variety.
Update docs to match the seeded dataset and verify build.
Bug fix prompt
Fix this runtime issue: [paste error].
Find root cause, patch only what is needed, validate, and explain why the fix works.
Customer collateral prompt
Create a customer-friendly how-to based on what we built.
Audience: business + developer stakeholders.
Output: clear steps, glossary, rollout plan, and next actions.

4. Definition of done

Before a task is complete

  • The app runs without runtime errors.
  • The build succeeds.
  • The user-facing flow is easy to follow.
  • The docs match the current behavior.
  • Another developer can continue from the notes you leave behind.

Common mistakes to avoid

  • Using vague prompts with no clear scope.
  • Making large changes without validating between steps.
  • Updating code but forgetting the docs.
  • Sending technical detail to the business without business context.

5. Recommended team operating model

For the first two or three weeks, use a two-person pairing model.

  • Driver: works in VS Code with Copilot and implements the change.
  • Reviewer: checks business alignment, tests behavior, and improves the prompts.

Suggested daily rhythm

  • 15 minutes: choose one business outcome.
  • 60 to 90 minutes: implement one small vertical slice.
  • 15 minutes: demo it and capture follow-ups.

6. First-week starter plan

Day 1Local setup, environment validation, run the current app, and read the engagement guide end-to-end.
Day 2Make one low-risk UI improvement and capture before-and-after screenshots.
Day 3Add one data or scoring enhancement and update docs to reflect the change.
Day 4Improve one operational workflow such as notifications, exception handling, or scheduling UX.
Day 5Dry-run a customer demo and finalize next-sprint recommendations.

7. What to share with business partners every sprint

  • What changed, in plain language.
  • Why it matters to the business, such as OTIF, cycle time, manual effort, or service level.
  • Evidence, such as screenshots, demo steps, and metrics.
  • Decision requests for the next sprint.
