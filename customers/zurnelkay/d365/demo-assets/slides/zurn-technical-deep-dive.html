Integration Patterns — ERP ↔ D365 Customer Service

Priority #2

⚡ Pattern 1: Dual-Write D365 F&O Only

  • Latency: Near real-time, bidirectional, synchronous
  • Infrastructure: Play/pause/catchup modes, online + offline support, combined error logs
  • Application scenarios (14 per MS Learn):
    • Integrated customer master & vendor master
    • Unified product mastering, pricing engine
    • Prospect-to-cash, procure-to-pay
    • On-hand inventory visibility
    • Project-to-cash, finance & tax reference data
  • Key: No-code/low-code — extend via table mapping
  • ⚠️ Native to F&O only — for SAP, use patterns 2-4

🔍 Pattern 2: Virtual Tables Any ERP

  • Latency: On-demand, per query
  • Full CRUD — not just read-only (create, read, update, delete)
  • External data projected as Dataverse tables WITHOUT replication
  • Supports OData endpoints, SQL, custom providers
  • Works with Power Automate, model-driven apps, Copilot
  • Best for: Order status, inventory check, warranty lookup
  • Agent sees ERP data inline on case form — no swivel-chair

🔄 Pattern 3: Power Automate Flows 900+ Connectors

  • Latency: Event-driven (seconds to minutes)
  • SAP connectors: SAP ERP connector (RFC & BAPI), SAP ODATA, certified
  • Triggers: Case created → create ERP service order; warranty claim → SAP RMA
  • Low-code — business users can modify flows
  • Built-in retry logic, error handling, approval chains
  • Premium connectors require per-user or per-flow license

🏭 Pattern 4: Azure Integration Services Enterprise

  • Latency: Configurable (real-time to batch)
  • Components: Azure Service Bus + Logic Apps + API Management + Functions
  • SAP-specific: Logic Apps SAP adapter (certified), supports RFC, BAPI, IDoc
  • Enterprise patterns: pub/sub, saga, dead-letter, guaranteed delivery
  • Full observability via Application Insights + Azure Monitor
  • Best for: High-volume, complex transformations, bi-directional SAP sync
💎 Zurn Elkay Recommendation: Since you're on SAP (not F&O), dual-write doesn't apply natively. Recommended hybrid: Virtual Tables for real-time lookups (orders, inventory) + Power Automate for event-driven flows (case → service order) + Azure Service Bus for high-volume batch sync (product catalog, pricing).

Voice Channel Architecture — Deployment Approaches

Priority #3

📞 Approach 1: Bring Your Own Telephony Most Enterprises

ACS Direct Routing or Teams Direct Routing

  • Connect SIP trunks via Azure Communication Services or Teams
  • Benefits: Reuse existing carriers & numbers, control call flows, preserve SIP context (caller ID, intent, auth tokens)
  • SBC required: Certified Session Border Controller connects your carrier to Azure
  • SLA: Microsoft responsibility starts after call enters ACS/Teams; outbound SIP is your responsibility
  • Best for: Organizations with existing telephony investments, SIP-dependent workflows (blind transfers, callbacks)
  • E.164 encoding required for phone numbers

☁️ Approach 2: Microsoft Calling Plan Simplest

  • Microsoft-managed PSTN numbers via ACS direct offer or Teams Calling Plan
  • Benefits: Simplified provisioning, billing, compliance; global reach with SLA
  • Consideration: SIP context (caller ID, intent) may be stripped by PSTN carriers outside Microsoft network
  • Best for: Cloud-first businesses, use cases not requiring full SIP metadata
  • Available countries: check ACS availability matrix per region

🔗 Approach 3: Teams Phone Integration New

  • Sync existing Teams Phone numbers directly into D365 Contact Center
  • PSTN options: Teams Calling Plan (service numbers only), Operator Connect, or Direct Routing
  • Requires: Enterprise Voice license, TeamsResourceAccount.Read.All Graph permission
  • Simplifies migration for orgs already on Teams Phone

🎙️ Voice Channel Capabilities

  • Codecs: G.711 and Opus (VoIP + PSTN)
  • Real-time transcription (30+ languages)
  • Live sentiment analysis during call
  • Call recording with PCI DSS compliance — pause/resume recording to avoid capturing sensitive data
  • IVR with Copilot Studio AI agents
  • Supervisor: monitor / whisper / barge
  • Voicemail → automatic case creation
  • Outbound dialing, click-to-call, callback
💎 Zurn Elkay: Most likely path → Approach 1 (Direct Routing) to retain your existing carrier & numbers. Can pilot with one queue while legacy handles the rest. SBC splits traffic by routing rules.

Unified Routing — Two-Stage Engine MS Learn

Priority #4

Stage 1: Classification

Add information to the work item

  • Rule-based: IF Brand = Zurn AND Product = Drains → set skills, priority
  • ML model (Intelligent Skill Finder): AI Builder category classification model trained on historical data
  • ML predicts: required skills, priority, product line, brand
  • AI reads email/chat body to determine intent automatically
  • Context variables set for downstream routing
  • Both rule-based and ML can run together — ML enriches, rules override
→

Stage 2: Assignment

Prioritize → then assign to best agent

  • Step 1 — Prioritize: Work items ranked by urgency, customer tier, wait time
  • Step 2 — Match: Agent skills (exact match or proficiency-based), capacity, presence
  • Assignment modes: Highest capacity, round-robin, or custom
  • Dynamic matching: Skills matched against agent attributes in real time
  • Overflow rules if no agent available (backup queue, relax skills, callback)
  • Works across ALL channels — truly omnichannel routing

🎯 Skills-Based Routing — Zurn Elkay Design

SkillTypeHow It's Set

Product: Drains / Sinks / FountainsProficiency (1-10)ML from case text or customer selection
Language: SpanishExact MatchCustomer language preference or detection

⏱️ SLA Engine Architecture

  • SLA KPIs: First Response (240 min), Resolution (480 min) per Zurn Elkay config
  • Warning at 75%: Email supervisor, boost priority
  • Breach at 100%: Auto-escalate to Tier 2, create task, notify manager
  • Pause: Waiting on Customer, On Hold — SLA timer pauses
  • Business hours: Per-queue calendar (8-5 M-F per Zurn Elkay config)
  • Visual timer: Countdown on case form — green → amber → red

Overflow, Escalation & Queue Design

Priority #4

🚨 Overflow Handling

  • Queue capacity limit: Max items before overflow triggers
  • Wait time threshold: If wait > X minutes → overflow
  • Actions available:
    • Route to backup queue
    • Relax skill requirements (widen agent pool)
    • Offer callback (voice)
    • Route to voicemail
    • Transfer to external number

⬆️ Escalation Patterns

  • SLA-driven: Auto-escalate on SLA warning/breach
  • Agent-initiated: Transfer to Tier 2 queue or supervisor
  • Hot-word triggered: ML detects escalation signals in conversation
  • Sentiment-driven: Negative sentiment sustained → alert supervisor
